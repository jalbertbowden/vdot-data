Description:
Abstract:
This feature class consists of approximately 125,000 features representing over 70,000 miles of Interstate, Primary, Secondary and Urban roads throughout the State of Virginia. The Linear Referencing System is based on the Virginia Department of Transportation's Source System of Record for road inventory, Highway Traffic Record and Inventory System (HTRIS).
Geometry and Attribution: The Linear Referencing System (LRS) data contained within this feature class provides dissolved route segmentation (i.e. routes are not segmented when they intersect other routes), thus rendering one table record per route. Incases where routes are noncontiguous (e.g. a valid physical gap exists), multi-part geometry is created. The feature class also depicts overlapping routes (e.g. Interstate 64 and Interstate 95) as two separate collinear features, one on top of the other. Routes built in the prime and non-prime directions are included. Each road centerline record has a master route record assigned. This feature class is best suited for labeling roadways when overlap reference is needed.
Measures: The linear reference is based on Official State Mileage (OSM) as derived from reference points at Highway Traffic Record and Inventory System (HTRIS) roadway intersections (i.e. nodes/junctions). Note: The overlap and non-prime measures are for reference only and have not been fully validated for accuracy or completeness.
Purpose:
This linear referenced data layer represents roadways that are maintained by the Virginia Department of Transportation and provides the underlying spatially enabled geometric network to which all "events" (e.g. potholes, pavement type, vehicle accidents, traffic counts, culverts, etc...) can be located.

Data_Quality_Information:
Attribute_Accuracy:
Attribute_Accuracy_Report:
Business attribute components for all Virginia Department of Transportation maintained routes are derived and maintained within Highway Traffic Record and Inventory System (HTRIS). The attributes in Highway Traffic Record and Inventory System (HTRIS) are field checked when a route is first created by the Maintenance Division or when modifications to existing routes are identified. The Roadway Network System continually reviews business processes and works with the business units to ensure the proper attribution is maintained as changes occur. Mechanisms developed for the business unit allows them to identify attribute incontinences if they should occur.
Logical_Consistency_Report:
The Linear Referencing System build process analyzes constructed routes determining if they are topologically continuous. Internal business rules dictate all necessary deviation from the aforementioned determination.
Completeness_Report:
The Highway Traffic Record and Inventory System (HTRIS) is the source system of record for this layer. The Roadway Network System (RNS) will replace Highway Traffic Record and Inventory System (HTRIS) and become the source system of record for this layer. The Highway Traffic Record and Inventory System (HTRIS) source data is split into two components, a business attribute component and a location component. Business Component: The business attribute component contains all records in the Highway Traffic Record and Inventory System (HTRIS). Location Component: There is currently a lag between the time a roadway is physically constructed and when it is entered into Highway Traffic Record and Inventory System (HTRIS) and then ultimately depicted the Geographic Information System Centerline. Computer-Aided Design files and local data are used where necessary. Ortho-rectified imagery is used in the creation and update process to ensure alignment. The Roadway Network System models Highway Traffic Record and Inventory System (HTRIS) and contains all location information, additional non-prime centerlines for divided primary and secondary routes and urban and street named networks. Currently the Roadway Network System contains most Highway Traffic Record and Inventory System (HTRIS) routes with a high degree of completeness for each route (95%+). Comparisons between the Highway Traffic Record and Inventory System (HTRIS) and the Roadway Network System networks will attempt to address all inconsistencies before the Roadway Network System becomes the source system of record. The Roadway Network System is based on creating a measured anchor section for each segment of the route. Detailed comparisons were done in the automated data conversion processes between the Highway Traffic Record and Inventory System (HTRIS) Link and Node based system and the Roadway Network System Anchor Section and Anchor Point based system to ensure completeness. Manual steps are being developed to handle special cases such as loop and loop back routes that were not possible to automate.
Positional_Accuracy:
Horizontal_Positional_Accuracy:
Horizontal_Positional_Accuracy_Report:
Horizontal accuracy for this layer can be defined as the probable error as measured along the linearly referenced route between reference calibration points. Such errors are largely dependent upon the degree of polyline resolution and the spatial alignment of the roadway with the actual centerline. Known reference point locations with known Official State Mileage values are used to ensure calibration of routes.
Quantitative_Horizontal_Positional_Accuracy_Assessment:
Horizontal_Positional_Accuracy_Value: 53 feet +/- 0.01 miles
Horizontal_Positional_Accuracy_Explanation:
Precision of the mile points in the Highway Traffic Record and Inventory System (HTRIS) is 53 feet (+/- 0.01 miles). Conversely location information may not have been recorded in the source system of record accurately such that errors of >0.1 miles can exist.

Process_Description:
Overview of geoprocessing Route Change Management: I. Get the GISE Road source feature class with the latest edits II. Get the road edges for the changed routes a. After this process is run - Then need to run calc_console.exe to set the 0-1 measures III. Delete then append edges by edge-rte-key to OSM and MPST change feature layers IV. Create and calibrate Interstate routes with the Mile Post measure points for changed routes a. After this process run SQL to update these measures back to the edge features V. Build edge-rte-key mile post measured features for changed routes VI. Build the official state measure (OSM) and mile post measure (MPST) changed route features VII. Append spatial data view based changed routes to full set for OSM and MPST layers VIII. Append spatial data view based change edge-rte-key to full set for OSM and MPST layers
